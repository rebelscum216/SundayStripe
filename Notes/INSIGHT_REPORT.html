01–06  Hidden Data — Already in DB, Never Shown

6 proposals · All implementable now

These columns are collected every sync cycle and sitting in Postgres. Zero new engineering required — just query and render. These are the fastest, cheapest insights to unlock.

P01
Refund/void exclusion from revenue calculations
orders · financialStatus column · all revenue displays
High Claude

Current problem

The orders.financialStatus column is populated with refunded, voided, authorized, and paid on every Shopify orders sync. However, all four statuses are summed equally into revenue totals. A refunded order inflates revenue figures. The revenue trend delta, the product detail revenue panel, the cross-channel table, and the inventory days-of-cover calculation all share this flaw.

Impact

  • Revenue numbers shown to the user are higher than actual net revenue
  • Days-of-cover calculations overstate sales velocity
  • Cross-channel opportunity flags may promote products that are being returned heavily
  • AI opportunity explainer receives inflated revenue context

Proposed fix — Claude

  • Add WHERE financial_status IN ('paid', 'partially_refunded') to every revenue query in app.controller.ts — affects /api/revenue, /api/revenue-trend, /api/products/:id/revenue, /api/inventory, /api/cross-channel, and /api/amazon/sales
  • Add a separate "Refunds (90d)" metric to the dashboard KPI strip sourced from financialStatus = 'refunded' rows
  • No schema change needed — data is already there
P02
Surface listing freshness — "Last seen" per channel listing
channelListings · lastSeenAt column · product detail page
High Claude

Current problem

The channelListings.lastSeenAt timestamp is updated on every sync but never rendered anywhere in the UI. A listing that hasn't been seen in 7 days may indicate a sync failure or a delisted product, but the user has no way to detect this without checking the operations page manually.

Impact

  • Silent listing disappearances go unnoticed until a customer can't buy
  • Product detail shows "published" status even if data is stale
  • No early warning before a sync gap grows into a data quality issue

Proposed fix — Claude

  • Add lastSeenAt to the /api/products/:id response (already joined, just not selected)
  • In the product detail page variant listing table, add a "Synced" column that shows relative time (e.g., "2 hours ago") with a yellow warning if > 24 hours and red if > 72 hours
  • Add buyabilityStatus to the same column — distinguish "published but not buyable" from "fully live"
P03
Show variant weight — unlock shipping cost estimates
variants · weightGrams column · product detail + inventory pages
Medium Claude

Current problem

variants.weightGrams is fetched from Shopify and stored on every product sync, but is never selected in any query or displayed in any page. It's invisible to the user. For products without weight set, weightGrams will be null — this itself is worth flagging since Merchant Center requires weight for some product types.

Impact

  • No way to estimate shipping costs per SKU (blocks margin analysis)
  • Missing weight can cause Merchant Center disapprovals for certain product types — no UI alert exists for this
  • Amazon requires accurate weight for fulfillment — unverified in current flow

Proposed fix — Claude

  • Add weightGrams to /api/products/:id variant rows (one-line select addition)
  • Display weight as "Xg / X lb" in the variant table on the product detail page
  • Add a missing_weight alert category to the missing_attribute checks — flag products where weightGrams IS NULL for Amazon/Merchant listings
P04
Display source freshness — when was this product last updated in Shopify?
products · sourceUpdatedAt column · product detail page
Medium Claude

Current problem

products.sourceUpdatedAt records when Shopify last reported an update to this product, but it's never shown. A product that hasn't been touched in Shopify for 2 years likely has stale copy, stale SEO, and stale images — but the user can't see this at a glance.

Impact

  • Users can't triage which products need attention most urgently
  • AI copy generation is more valuable when the product data is known to be stale
  • No way to sort or filter the product catalog by "most stale"

Proposed fix — Claude

  • Expose sourceUpdatedAt on /api/products/:id and /api/products list endpoint
  • Show "Source last updated: X days ago" in the product detail header, styled yellow if > 180 days, red if > 365 days
  • Add "Last updated" as a sortable column option in the product list table
P05
AI recommendation accept/dismiss rate — measure what's working
aiRecommendations · acceptedAt, dismissedAt · new insights panel
Medium Claude

Current problem

aiRecommendations.acceptedAt and dismissedAt are set when the user takes action, but there's no UI that aggregates this into an acceptance rate. The system has no feedback loop — it can't self-improve or show the user which AI features are delivering value.

Impact

  • No signal on whether AI suggestions are helpful or being ignored
  • Can't compare acceptance rate by sourceType (SEO vs. Amazon vs. product copy)

Proposed fix — Claude

  • Add a "Previously accepted" panel on product detail page that lists all accepted recommendations for that product with timestamps
P06
Non-buyable listing alert — separate "published but can't sell" from healthy
channelListings · buyabilityStatus column · alerts + product list
High Claude

Current problem

channelListings.buyabilityStatus stores "not_buyable" for listings that appear published but can't actually be purchased (Merchant Center disapprovals that still show in search, Amazon suppressed offers). The product list and cross-channel table treat status = "published" as fully healthy without checking buyability.

Impact

  • A product can appear "live" on all channels while generating zero revenue — silently
  • Cross-channel table may flag "no_revenue" without explaining the root cause (unbuyable listing)
  • No alert is auto-generated for not_buyable listings that slip through

Proposed fix — Claude

  • In merchant-sync.service.ts and amazon-sync.service.ts, auto-create a listing_issue alert when buyabilityStatus flips to not_buyable
  • In cross-channel table, add a suppressed flag alongside no_listing so users can filter and act on it

07–11  Apply Flow Gaps — Channels Missing Write-Back

5 proposals

Shopify has full two-way coverage. Amazon has partial attribute write-back. Merchant Center and GSC are read-only. These proposals extend the "view → fix → push" loop across the remaining gaps.

P07
One-click "Create Amazon Listing" from cross-channel table
cross-channel page · POST /api/amazon/import-listing · already exists
High Claude

Current problem

When a product shows the no_listing flag in the cross-channel table (meaning it's on Shopify/Merchant but not Amazon), the user must navigate away to the product detail page to create a listing. POST /api/amazon/import-listing already exists. The cross-channel table's AI explainer fires but doesn't surface a direct action button.

Impact

  • Key revenue opportunity: products not on Amazon represent unaddressed demand
  • Current UX requires 3 clicks + manual navigation where 1 click should suffice
  • The AI explainer identifies the opportunity but can't close the loop

Proposed fix — Claude

  • In cross-channel-table.tsx, add a "List on Amazon →" button in the expanded AI panel when flag === 'no_listing' and Amazon integration is connected
  • Button calls POST /api-proxy/amazon/import-listing with the product's SKU as the candidate; show a loading state then success/error toast
  • After success, refresh the row to show the new listing status
  • Include current inventory count in the expanded panel so user knows if there's stock to sell before creating the listing
P08
Merchant Center — push product data updates via Shopify feed sync signal
Merchant Center apply flow · currently read-only
High Codex

Current problem

There is no PATCH endpoint for Merchant Center. All Merchant data is read-only. When a user fixes a missing attribute alert (e.g., missing brand, missing description) via the Shopify write-back, the Merchant Center listing won't reflect the fix until the next GMC feed crawl (up to 24 hours). The user has no way to force-trigger a re-crawl or verify the fix propagated.

Impact

  • Fixes applied to Shopify sit in limbo until Google re-crawls — no confirmation
  • Listing may remain disapproved for 24+ hours after the root cause is resolved
  • No "re-submit for review" action means users can't accelerate fix verification

Proposed fix — Codex

  • After any PATCH /api/products/:id/attributes or /seo call, enqueue a merchant-sync job scoped to that product's listing ID using products.requestReview() from the Content API v2.1
  • Display a "Resubmitting to Merchant Center…" status badge on the product detail channel listing row after a fix is applied
  • Implement POST /api/merchant/listings/:listingId/resubmit endpoint in app.controller.ts that calls Content.products.insert() with the updated product data (requires products write scope)
  • Add write_products scope request to the Merchant OAuth setup flow
P09
Amazon price visibility + price update endpoint
Amazon apply flow · no price read or write currently
High Codex

Current problem

The Amazon sync fetches listing attributes but does not fetch or store the current Amazon sale price. There is no amazonPrice field in the schema. The cross-channel table and product detail page show no Amazon price — making it impossible to detect price disparity between Shopify and Amazon without leaving the app. Price mismatch is a top source of Amazon suppressed listings.

Impact

  • Suppressed listings from price parity violations are invisible in the current alert system
  • No way to manage Amazon pricing strategy from the hub
  • A common seller pain: "why is my Amazon listing suppressed?" — often price-related but our system can't diagnose it

Proposed fix — Codex

  • Add listingPrice (integer cents) and currency columns to channelListings (new migration)
  • Pull price from the Amazon Listings Items API summaries[].buyingPrice.listingPrice during sync and upsert into the new column
  • Add PATCH /api/products/:id/amazon-price endpoint using Listings.patchListingsItem() with attribute purchasable_offer
  • Surface current Amazon price vs. Shopify price in the product detail variant table — highlight in yellow if delta > 5%
  • Auto-generate a price_mismatch alert when the two prices diverge by more than a configurable threshold
P10
Bulk attribute apply — fix missing fields across multiple products at once
products list page · bulk PATCH flow · currently one-at-a-time only
Medium Codex

Current problem

All apply flows are single-product. If 30 products are missing a brand name, the user must navigate to each product detail page individually, enter the brand, and click Apply. The product list page shows which products have gaps but provides no way to act on multiple at once.

Impact

  • Fixing common attribute gaps (brand, weight, GTIN) for 50+ products takes hours of manual work
  • Merchant Center and Amazon listings remain disapproved during this window
  • High-friction UX discourages users from maintaining data quality

Proposed fix — Codex

  • Add a checkbox column to the products list table for multi-select
  • Add a POST /api/products/bulk-update endpoint that accepts an array of {id, brand, ...} objects and calls the individual PATCH handlers in sequence (or in parallel with rate limiting)
  • Show a floating action bar at the bottom of the screen when >1 product is selected: "Set brand for X products", "Mark GTIN exempt for X products"
  • Apply each change to Shopify and queue downstream sync jobs, returning a summary of successes/failures
P11
Alert auto-resolve — when a fix is applied, close the linked alert automatically
alerts system · PATCH /api/alerts/:id/resolve · currently manual
Medium Claude

Current problem

When a user applies a fix for a missing attribute (e.g., adds a barcode via the variant fix UI), the underlying alert remains open. The user must manually navigate to the alerts page and resolve it. The two workflows (apply fix vs. resolve alert) are completely disconnected.

Impact

  • Alert count stays artificially high even after work is done — discourages users
  • Alert triage may re-surface already-fixed issues in AI groupings
  • No audit trail linking "alert X was resolved by applying fix Y on date Z"

Proposed fix — Claude

  • In each PATCH endpoint that applies a fix (/seo, /attributes, /variants/:id/barcode, /gtin-exempt, /amazon-attribute), after a successful push, query for open alerts matching the product ID and the relevant category/entityRef and mark them resolved
  • Add an optional resolvedBy text column to alerts (new migration) so the resolution method is recorded: "user_apply_barcode", "user_apply_seo", etc.
  • Show a green "1 alert auto-resolved" toast after a successful fix application

12–16  Product Catalog & Detail Improvements

5 proposals

The product catalog is the operational core of the hub. These proposals add missing context columns, revenue-informed sorting, and richer product detail interactions.

P12
Revenue + GSC columns in product list — show which products matter most
products/page.tsx · /api/products endpoint · join orders + searchPerformance
High Codex

Current problem

The products list shows inventory, barcode, and description status but has no revenue or traffic column. The user cannot sort by "highest revenue" or "most organic traffic" to prioritize which products to fix first. Every product row looks equally important even if some drive 80% of revenue.

Impact

  • Users spend time fixing low-revenue products while high-revenue products have issues that go unaddressed
  • The products list is a triage surface but provides no triage signals
  • The cross-channel page has revenue data but isn't designed as a fix-action surface

Proposed fix — Codex

  • Add a ?include=revenue,gsc query param to GET /api/products that performs a LEFT JOIN to order_line_items (sum unitPriceCents * quantity per product, 90d, paid/partially_refunded only) and a second JOIN to search_performance (sum clicks by product URL, 90d)
  • Add "Revenue (90d)" and "GSC Clicks (90d)" as optional columns with a column picker in the product list UI — hidden by default for speed, enabled by toggle
  • Add "Sort by Revenue ↓", "Sort by Alerts ↓", "Sort by Quality Score ↓" to the existing sort controls
  • For stores > 200 products, implement cursor-based pagination in place of the current hardcoded LIMIT 200
P13
"Best seller" variant badge on product detail
products/[id]/page.tsx · /api/products/:id/revenue · already have the data

The product detail page fetches revenue by variant from /api/products/:id/revenue and displays it, but doesn't highlight which variant is the top seller. All variants appear equal in the listing table above. The user has to mentally cross-reference the revenue panel with the variant table.

Impact

  • When fixing variants (barcode, attributes, Amazon rewrite), users should prioritize the best-selling size/color first
  • A missing barcode on a top-selling variant is more urgent than on a slow-moving one

Proposed fix — Claude

  • Add a small "⭐ Top seller" badge next to that variant's SKU in the variant listing table

P14
Inline description editor with direct Shopify push
products/[id]/page.tsx · PATCH /api/products/:id/attributes · descriptionHtml not editable
Medium Claude

Current problem

PATCH /api/products/:id/attributes already accepts and pushes descriptionHtml to Shopify, but there is no UI to edit it. Users can only view the current description. The "describe product" AI action generates a description but the user can't see a diff or paste it in — they have to copy the AI output and go to Shopify admin manually.

Impact

  • The AI description generator is the most frequently-used AI action but the apply step requires leaving the hub
  • Missing or thin descriptions are a top Merchant Center disapproval reason — there's no in-app fix path

Proposed fix — Claude

  • Add "Apply AI description" button when an aiRecommendations record of sourceType='product' exists for this product — pre-populates the textarea with the AI text
  • On save, call PATCH /api-proxy/products/:id/attributes with {descriptionHtml: value} — the endpoint already handles the Shopify push
  • Show a diff view (old vs. new) before confirming
P15
Inventory days-of-cover alert — auto-flag stockout risk
/api/inventory endpoint · inventoryPositions · currently informational only
High Codex

Current problem

The inventory page calculates days-of-cover per variant (available qty ÷ daily sales velocity) and shows status flags (out_of_stock, stock_risk, low_stock). However, this calculation is only visible on the inventory page — it never generates an alert, never surfaces in the dashboard AI actions, and never appears on the product detail page for affected variants.

Impact

  • Stockouts are the single most damaging event for a Shopify/Amazon seller — lost sales, ranking drops, listing suppression
  • The system already has all the data to predict stockouts 7–14 days out but doesn't act on it
  • High-revenue variants going out of stock undetected is a major revenue risk

Proposed fix — Codex

  • Add a nightly BullMQ job (inventory-alert queue) that runs the days-of-cover calculation and creates severity: 'high' alerts of category: 'stock_risk' for any variant where daysOfCover < 14 and revenue > $0 in last 90d
  • Add category: 'out_of_stock' alerts for variants at zero qty with recent sales
  • Surface these alerts in the dashboard AI action feed with a "Reorder now" CTA (external link to supplier or Shopify purchase orders)
  • De-duplicate: only create a new alert if no open alert for the same variant already exists
P16
Product search by title/SKU on catalog page
products/page.tsx · /api/products already supports ilike · UI doesn't expose it
Medium Claude

Current problem

The NestJS products endpoint supports an ilike search filter, but the product list UI has no search input. For catalogs with 100+ products, the user must visually scan the table to find a specific item. The table defaults to alphabetical order by title with no ability to filter.

Impact

  • Finding a specific product to investigate is slow and error-prone at scale
  • A 200-product limit compounds this — important products may be hidden beyond the cut-off

Proposed fix — Claude

  • Add a search input to the product list page header that debounces (300ms) and appends ?search= to the /api/products request
  • In app.controller.ts, confirm the existing ilike filter covers both title and canonicalSku fields
  • Show a result count ("Showing 12 of 184 products") below the search input

17–20  Alert System Upgrades

4 proposals

The alert system is functional but reactive. These proposals add bulk actions, alert snoozing, SLA tracking, and a richer payload parser to make alert triage faster and more actionable.

P17
Bulk alert resolve — dismiss entire categories at once
alerts/page.tsx · PATCH /api/alerts/:id/resolve · one-at-a-time only
Medium Claude

Current problem

Alerts can only be resolved one at a time via individual PATCH calls. After a sync that generates 50 missing_attribute alerts for the same issue (e.g., brand missing), the user must click resolve 50 times. There is no "resolve all in this category" or "select multiple + resolve" action.

Impact

  • Discourages users from keeping alerts clean — leads to alert fatigue
  • The AI triage panel groups by root cause but can't resolve the group

Proposed fix — Claude

  • Add POST /api/alerts/bulk-resolve accepting an array of alert IDs — sets all to resolved in a single transaction
  • In the alerts page, add checkboxes to each alert row and a "Resolve selected (N)" button in the table header
  • Add a "Resolve all in group" button to each severity/category group header
  • In the AI triage panel, after a priority group is shown, add a "Resolve all N alerts in this group" action that calls the bulk endpoint
P18
Alert snooze / suppression — silence recurring noise
alerts schema · new snoozedUntil column + snooze UI
Medium Codex

Current problem

Recurring alerts that can't be immediately fixed (e.g., a supplier hasn't provided GTINs yet) resurface every sync cycle. There is no snooze or suppression mechanism. The only option is to resolve the alert knowing it will re-open the next day, degrading the alert system's signal-to-noise ratio.

Impact

  • Users stop checking alerts because they're full of known, unfixable issues
  • Critical new alerts get buried under chronic recurring noise
  • No way to "park" an alert while waiting for a vendor response

Proposed fix — Codex

  • Add snoozedUntil (timestamp, nullable) to the alerts table (new migration)
  • Update sync services to skip creating a duplicate alert if an existing alert with the same entityRef + category is snoozed and snoozedUntil > NOW()
  • Add a "Snooze" dropdown button on each alert row: "1 day", "7 days", "30 days", "Until I resolve"
  • Add a "Snoozed" tab to the alerts page (alongside Open/Resolved) so snoozed alerts remain visible but separate
P19
Alert age display — show how long each issue has been open
alerts/page.tsx · alerts.createdAt · not currently displayed
Low Claude

Current problem

alerts.createdAt is stored but never rendered in the alerts table. The user cannot tell if an alert is 2 hours old or 6 months old. A "listing issue" that's been open for 90 days is far more urgent than one from this morning, but both appear the same.

Impact

  • Chronic issues look the same as fresh ones — no urgency signal from age
  • SLA-conscious operators can't track mean-time-to-resolve without this data

Proposed fix — Claude

  • Add an "Age" column to the alert table (e.g., "3 days", "2 months") derived from createdAt — already returned by the API
  • Style age in yellow if > 7 days, red if > 30 days
  • Add "Sort by Age ↓" to make it easy to find the oldest chronic issues
P20
Rich alert payload parser — normalize Merchant vs. Amazon issue formats
alerts/page.tsx · alerts.payloadJson · inconsistent rendering
Medium Codex

Current problem

alerts.payloadJson structure varies by source platform. Merchant Center alerts include servability, attributeName, description. Amazon alerts include severity, code, message. The current UI tries to display these generically, producing inconsistent and sometimes unreadable alert descriptions. The AI explainer compensates, but it costs a GPT call every time.

Impact

  • Users can't understand alerts without clicking "Explain with AI" — which should be optional, not required for basic comprehension
  • Misrendered payloads erode trust in the alerts system
  • Every AI explain call has a cost; normalizing payloads would reduce unnecessary GPT usage by ~30%

Proposed fix — Codex

  • Create a server-side normalizeAlertPayload(sourcePlatform, payloadJson) utility in app.controller.ts that returns a consistent {title, description, affectedAttribute, suggestedAction} shape
  • Map known Merchant Center codes (MISSING_ATTRIBUTE, INVALID_VALUE, IMAGE_TOO_SMALL) and Amazon codes (INVALID_ATTRIBUTE, MISSING_REQUIRED) to human-readable templates
  • Return this normalized shape on GET /api/alerts alongside the raw payloadJson
  • Replace the current fragile UI payload render with the normalized fields — reserve AI explainer for complex/unknown codes

21–23  Search Console & SEO Enhancements

3 proposals

The GSC dashboard is the most data-rich page in the hub. These proposals add trend context, branded/non-branded segmentation, and CSV export — converting it from a snapshot into a decision engine.

P21
Branded vs. non-branded query segmentation
search-console/page.tsx · searchPerformance table · no segmentation today
High Codex

Current problem

The GSC query table mixes branded queries (e.g., "your brand name shoes") with non-branded discovery queries. Branded clicks are loyalty traffic; non-branded clicks are growth. The AI quick-wins optimizer and almost-page-1 table currently include branded queries, which inflate opportunity counts and dilute the actual SEO growth potential.

Impact

  • SEO recommendations for branded terms are low-value (you already rank #1)
  • True growth opportunity is non-branded impressions at low CTR — currently mixed in
  • Investors/stakeholders want branded vs. non-branded split to evaluate brand equity vs. SEO performance

Proposed fix — Codex

  • Add a workspace.brandTerms JSON array field to the workspaces table (or a separate brand_terms config table) that the user populates with their brand name variants
  • In the GSC sync service, tag each searchPerformance row with isBranded: boolean based on whether dimensionValue contains any brand term (case-insensitive)
  • Add a "Branded / Non-Branded / All" toggle above the quick-wins table and query explorer — defaults to Non-Branded
  • Show separate KPI cards for branded clicks vs. non-branded clicks in the GSC summary strip
P22
GSC data window expansion + period-over-period trend
searchPerformance · periodDays hardcoded to 90 · no historical trend
Medium Codex

Current problem

Every GSC sync replaces the previous data with a fresh 90-day window (periodDays = 90 always). There is no rolling history — the system can't answer "are clicks improving or declining month over month?" The fetchedAt timestamp exists but historical snapshots are overwritten each sync cycle.

Impact

  • SEO progress is invisible — after implementing a meta title change, users can't measure the click improvement
  • Seasonal patterns can't be identified (e.g., holiday traffic spikes vs. normal baseline)
  • No way to validate that the "Apply to Shopify" quick-wins flow is actually improving CTR over time

Proposed fix — Codex

  • Add a snapshotDate column to searchPerformance (date type, nullable, new migration) — when set, this row is a historical snapshot and is not overwritten by future syncs
  • Modify gsc-sync.service.ts: before upserting the new 90-day data, save a daily snapshot of total clicks/impressions/position aggregated at the workspace level into a new gsc_daily_snapshots table
  • Add a sparkline chart to the GSC summary KPI cards showing the last 30 daily snapshots for clicks and impressions
  • After a quick-win SEO change is applied, record the date on the aiRecommendations row so the trend chart can overlay "SEO change applied" markers
P23
GSC data export to CSV
search-console/page.tsx · no export currently
Low Claude

Current problem

The GSC query and page tables render up to 100 rows each but provide no way to export the data. Users who want to share SEO reports with clients or do deeper analysis in a spreadsheet must manually copy data. The search performance data is all in Postgres — export is a one-line query.

Impact

  • Agency users managing multiple brands need exportable reports for client deliverables
  • Power users who want to run pivot analysis (e.g., by page type or product category) are blocked

Proposed fix — Claude

  • Add GET /api/search-console/export endpoint that returns all dimension = 'query' rows as Content-Type: text/csv with headers: Query, Clicks, Impressions, CTR%, Position
  • Add an "Export CSV" button above the query explorer table in the GSC page
  • The button triggers a browser download via a link pointing to the API export route (no client-side CSV generation needed)

24–27  Structural & Strategic Gaps

4 proposals

These are the largest missing dimensions — cost/margin data, COGS tracking, and a sync health dashboard. They require new schema and infrastructure but unlock an entirely new category of insight.

P24
Cost / COGS per SKU — the single biggest missing data dimension
No cost table exists · blocks margin, ROI, and channel profitability analysis
High Codex

Current problem

There is no cost-of-goods column anywhere in the schema. The hub knows revenue but not profit. Every AI recommendation (cross-channel opportunities, Amazon expansion, SEO prioritization) is based on revenue only — which is misleading if low-margin products dominate revenue. A product doing $50k/year at 2% margin is less worth optimizing than one doing $20k at 40% margin.

Impact

  • No margin calculation possible — the hub can't tell the user if their Amazon expansion will be profitable after fees
  • AI opportunity explainer recommends expanding listings that may actively lose money on Amazon FBA fees
  • Inventory days-of-cover is calculated by units, not by profit-at-risk — wrong prioritization
  • Cross-channel priority score has no profit weighting

Proposed fix — Codex

  • Add a product_costs table: (id, variantId, costCents, currency, effectiveFrom, source) — supports historical cost tracking and multiple currencies
  • Add a simple COGS entry UI in product detail: "Cost per unit: $___" input that writes to product_costs
  • Alternatively, pull cost from Shopify inventory items (inventoryItem.unitCost) — Shopify stores this if the merchant uses cost tracking; add to the Shopify sync GQL query
  • Compute grossMarginCents = (unitPriceCents - costCents) * quantity in all revenue queries and surface margin % alongside revenue in the product detail revenue panel, cross-channel table, and dashboard KPIs
  • Update AI context in cross-channel-opportunity and amazon-listing-rewrite endpoints to include margin data
P25
Sync health dashboard — per-integration job history, duration, and error log
operations/page.tsx · syncJobs table · currently shows failures only
Medium Codex

Current problem

The operations page shows failed sync jobs but no success history. The user can't see when the last successful Amazon sync ran, how long it took, or how many records were processed. syncJobs.payloadJson stores sync result summaries (e.g., {productsProcessed: 45, listingsUpdated: 12}) but this data is never displayed.

Impact

  • Users can't verify that their data is fresh — "Is my Merchant data from today or last week?"
  • retryCount is incremented but never capped or displayed — jobs may retry indefinitely
  • No way to detect a sync that's consistently timing out before it officially fails

Proposed fix — Codex

  • Add GET /api/operations/job-history endpoint returning the last 10 sync jobs per integration, ordered by startedAt DESC, including state, duration (finishedAt - startedAt), payloadJson summary, and retryCount
  • Replace the operations page's failure-only view with a full job history table with status icons (✓ done, ⚠ failed, ↻ running, ⏸ pending)

P26
Revenue attribution by channel — Shopify vs. Amazon revenue split
orders · integrationAccountId · split not surfaced in any dashboard
High Claude

Current problem

Orders are tagged with integrationAccountId which links back to a specific platform (Shopify, Amazon). However, all revenue displays aggregate across all channels. There is no "Shopify revenue vs. Amazon revenue" breakdown on the dashboard or any product detail view. GET /api/amazon/sales exists but is not wired to any UI component.

Impact

  • Sellers can't tell which channel is growing — they can't make informed decisions about where to invest marketing spend
  • Product detail revenue panel shows a single total with no channel context
  • The cross-channel opportunity score has no channel-specific revenue weighting

Proposed fix — Claude

  • Update /api/products/:id/revenue to group results by integrationAccountId → platform (join to integrationAccounts) and return a byChannel breakdown alongside the variant totals
  • On the product detail revenue panel, add a channel breakdown row: "Shopify: $X · Amazon: $Y" (or a small horizontal bar chart)
  • Wire the existing /api/amazon/sales endpoint to a new "Channel Split" KPI card on the dashboard that shows Shopify % vs. Amazon % of total revenue
  • No schema change needed — data is already attributed
P27
Workspace-level performance digest — weekly email/notification summary
New feature · BullMQ weekly job · digest email or in-app notification
Low Codex

Current problem

The hub is a pull-based tool — users must actively log in to discover new issues, new opportunities, and weekly performance trends. There is no push notification, webhook outbound, or email digest. A seller managing multiple platforms may not check the hub daily, meaning critical alerts and stock risks go unnoticed for days.

Impact

  • High-severity alerts that need same-day action sit unread until the user happens to log in
  • No retention mechanism — inactive users have no reminder that the tool is monitoring their stores
  • Revenue trends and opportunity counts are a compelling engagement hook if delivered proactively

Proposed fix — Codex

  • Add a weekly BullMQ cron job (digest-email queue, Monday 8am local timezone from workspace.timezone) that queries: new critical/high alerts count, revenue vs. prior week, top 3 SEO opportunities, top stockout risks
  • Send via SendGrid or Resend (add SENDGRID_API_KEY to env) to workspace.ownerEmail (add this field to workspaces table)
  • Alternatively, deliver as an in-app notification banner on next login: "You have 3 new critical alerts and 2 products approaching stockout since your last visit"
  • Include a one-click "Resolve all alerts" deep link in the email that auto-authenticates via a short-lived token
