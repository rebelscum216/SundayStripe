Sunday Stripe Architecture

A working map of the Sunday Stripe commerce operations hub: Shopify theme, Next.js dashboard, Nest API, BullMQ workers, Postgres schema, Redis queues, external channel integrations, and deployment surfaces on Vercel, Render, Railway, and local Docker.

Overview

Sunday Stripe is a multi-channel commerce operations system. The repository contains both the customer-facing Shopify theme and an internal hub that pulls data from Shopify, Google Search Console, Google Merchant Center, and Amazon SP-API. The hub normalizes channel data into Postgres, surfaces operational views in a Next.js app, and uses AI endpoints to explain issues, draft fixes, and produce channel-specific recommendations.

Primary Apps

apps/web Next.js 14 dashboard with React server components, server actions, and a dark sidebar shell.

apps/api NestJS API and worker host using BullMQ, Drizzle, and channel-specific services.

packages/db Shared Drizzle schema and migration package.

Commerce Surface

Shopify theme Root-level Liquid sections, snippets, assets, config, templates, and locales.

content Static article HTML used by publishing scripts.

dashboard Earlier Python dashboard prototype backed by data access modules.

The system is structured as a monorepo. The dashboard and API are the operational product, while the root Shopify theme is the storefront product.

The API process currently hosts both HTTP routes and queue processors. That means the deployed API container is not only a request server; it also runs scheduled jobs and background work when Redis is reachable.

Data Flow

1. Connection setup. Integration credentials are supplied through environment variables or OAuth flows and stored in integration_accounts when applicable.
2. Sync orchestration. Manual API calls, schedulers, OAuth callbacks, scripts, or webhooks enqueue BullMQ jobs.
3. Channel ingestion. Channel services call external APIs, normalize payloads, and upsert shared entities such as products, variants, listings, inventory, search performance, orders, and alerts.
4. Dashboard reads. Next.js server components fetch API endpoints with cache: "no-store" so pages reflect live operational data.
5. Actions and AI. Server actions and client proxy calls post mutations or AI requests to the Nest API. Some mutations update Postgres and enqueue follow-up sync jobs.
6. External writeback. Shopify SEO, product attributes, GTIN/barcodes, Merchant syncs, and Amazon import/listing operations are pushed back through the relevant channel API paths.

Next.js Web App

The web app lives in apps/web and uses Next.js App Router. The root layout builds a fixed application shell: DrawerProvider, SidebarNav, a scrollable main content region, and the global AiDrawer. Most pages are server-rendered and fetch the Nest API directly through API_BASE_URL.

Frontend responsibilities

  • Dashboard navigation and page composition.
  • Operational views for products, alerts, inventory, Search Console, Shopify, Merchant Center, Amazon, cross-channel opportunities, operations, and settings.
  • Server actions for AI generation, alert handling, sync triggers, failed job cleanup, SEO updates, and Amazon import flows.
  • Client-side proxy calls from interactive components that need browser-triggered API access without exposing the API origin in component code.

Client API Proxy

apps/web/app/api-proxy/[...path]/route.ts forwards browser-originated requests to the Nest API. It reconstructs the upstream URL as ${API_BASE_URL}/api/{path}, strips hop-by-hop headers, buffers non-GET bodies to set content length, disables caching, follows no redirects, and aborts upstream calls after 9 seconds.

This proxy is used by interactive components such as Search Console trend charts, product attribute fixes, Amazon import actions, and topbar search. It keeps the client code simple while letting Vercel host a same-origin route.

Nest API

The API lives in apps/api. main.ts creates a Nest app with raw body support, sets the global prefix to /api, enables CORS for WEB_APP_URL, and listens on PORT with a default of 3001.

Module graph

  • AppModule wires config, BullMQ Redis connection, scheduled jobs, shared queues, database, and channel modules.
  • DatabaseModule provides a postgres-js client and Drizzle database instance using DATABASE_URL.
  • ShopifyModule owns OAuth, webhooks, initial sync, order sync, and webhook processing.
  • MerchantModule owns Google Merchant Center API access, sync services, processor, and scheduler.
  • GscModule owns Google Search Console API access, sync services, processor, and scheduler.
  • AmazonModule owns Amazon SP-API access, listing sync, order sync, processor, and scheduler.
  • InventoryModule owns inventory alert generation and scheduled checks.

Queues and Schedulers

BullMQ is configured globally with REDIS_URL. The connection disables offline queueing, sets maxRetriesPerRequest to null, uses a 5 second connect timeout, and does not retry connection failures through BullMQ's retry strategy. This favors fast failure over hanging when Redis is unavailable.

Migrations

SQL migrations live in packages/db/drizzle. The root pnpm migrate script delegates to @sunday-stripe/db, which runs tsx migrate.ts.

Shopify

Shopify appears in two forms: the storefront theme at the repository root and the API integration inside apps/api/src/shopify.

Theme architecture

  • layout/theme.liquid and layout/password.liquid define top-level storefront shells.
  • sections contains product, collection, cart, article, search, banner, footer, header, and account sections.
  • snippets contains reusable Liquid fragments for product cards, media, price, variant pickers, cart UI, icons, localization, pagination, and menus.
  • assets contains CSS and JavaScript for cart, predictive search, product forms, galleries, localization, quick add, media, and base styling.
  • templates maps Shopify page types and custom pages to JSON section compositions.

Admin integration

  • OAuth starts at /api/shopify/auth?shop=... and returns through /api/shopify/callback.
  • The OAuth service validates HMAC/state, exchanges the auth code, stores integration data, and enqueues initial sync.
  • Webhooks post to /api/shopify/webhooks. Raw body HMAC verification happens before JSON parsing.
  • Verified webhook jobs are enqueued with the Shopify webhook id as the BullMQ job id for idempotency.
  • Sync services call the Shopify Admin GraphQL API and upsert products, variants, inventory, orders, and channel metadata.

Environment

Key variables include SHOPIFY_SHOP, SHOPIFY_API_KEY, SHOPIFY_API_SECRET, SHOPIFY_SCOPES, SHOPIFY_API_VERSION, SHOPIFY_APP_URL, and SHOPIFY_WEBHOOK_CALLBACK_URL.

Google Search and Merchant

Search Console

GscModule uses GscApiService, GscSyncService, GscSyncProcessor, and GscSchedulerService. GSC data feeds Search Console dashboard pages, almost-page-1 opportunities, low CTR tables, product page SEO opportunities, daily trends, branded query segmentation, and AI page optimization.

Merchant Center

MerchantModule uses Merchant API, sync service, processor, and scheduler classes. Merchant data feeds listing quality, buyability, product attributes, missing identifier flows, Merchant Center channel status, and product resync after attribute or SEO fixes.

Environment

Search Console uses GSC_SITE and GSC_TOKEN_JSON. Merchant Center uses GOOGLE_MERCHANT_ID, GOOGLE_MERCHANT_CREDENTIALS_JSON, and GOOGLE_MERCHANT_DEVELOPER_EMAIL.

Amazon SP-API

Amazon integration lives in apps/api/src/amazon. The API client handles Amazon auth and SP-API requests. Sync services ingest listings and orders. Dashboard surfaces include Amazon unmatched listings, sales, cross-channel opportunities, Amazon attribute suggestions, attribute fixes, import listing, and AI listing rewrite.

Environment

Key variables include AMAZON_SELLER_ID, AMAZON_CLIENT_ID, AMAZON_CLIENT_SECRET, AMAZON_REFRESH_TOKEN, AMAZON_MARKETPLACE_ID, AMAZON_REGION, AWS_ACCESS_KEY_ID, and AWS_SECRET_ACCESS_KEY. The default marketplace in render.yaml is ATVPDKIKX0DER and the default region is us-east-1.

Vercel

The Next.js dashboard is shaped for Vercel deployment. The web app has its own apps/web/vercel.json, while the repository root vercel.json is currently an empty object. The app uses App Router, route handlers, server components, and server actions, all of which map naturally to Vercel functions.

Important Vercel behavior

  • /api-proxy/* runs as a Vercel route handler and aborts upstream API calls after 9 seconds to fit short function limits.
  • /api/keep-alive calls ${API_BASE_URL}/api/status, likely to keep the API warm or provide a lightweight monitor.
  • The web app should set API_BASE_URL to the Render or Railway API origin.
  • The API should set WEB_APP_URL to the Vercel dashboard URL for CORS.

Render

render.yaml defines a Docker-backed web service named sunday-stripe-api. It uses apps/api/Dockerfile with repository root as the Docker context and checks health at /api/status. The service listens on PORT=3001.

Render environment groups by role

  • Runtime: NODE_ENV, PORT, WEB_APP_URL.
  • Persistence: DATABASE_URL for Neon/Postgres, REDIS_URL for Upstash/Redis.
  • Secrets: TOKEN_ENCRYPTION_KEY, OPENAI_API_KEY.
  • Shopify: shop domain, API credentials, scopes, API version, app URL, webhook callback URL.
  • Google: GSC token/site and Merchant Center service account configuration.
  • Amazon: Seller, LWA client credentials, refresh token, marketplace, region, AWS keys.

Railway

railway.json is an alternate API deployment target. It builds from apps/api/Dockerfile, uses /api/status for health checks, and restarts on failure up to three times.

Security Boundaries

  • CORS is limited by WEB_APP_URL, defaulting locally to http://localhost:3000.
  • Shopify webhooks require raw body HMAC verification before parsing or enqueueing.
  • Shopify OAuth validates signed callback parameters and state before storing integration credentials.
  • Token encryption is represented by TOKEN_ENCRYPTION_KEY and encrypted token fields in integration_accounts.
  • Server actions keep many privileged API mutations server-side in Next.js.
  • The browser proxy is same-origin but should still be treated as a pass-through to privileged API routes; production hardening should add authentication and authorization checks.

Observability

Observability is currently operational rather than platform-deep. The API has a health endpoint, queue failure listing, failed/pending cleanup endpoints, Nest loggers in processors/controllers, BullMQ job state, and persisted sync_jobs.error_json. Render and Railway health checks hit /api/status. Vercel can observe the web route handlers and server component fetches.

Useful operational paths

  • /operations in the web app for API status and failed jobs.
  • GET /api/operations/jobs/failed for queue failures.
  • DELETE /api/jobs/failed and DELETE /api/jobs/pending for cleanup.
  • sync_jobs table for durable sync state and error payloads.

Risks and Gaps

  • Authentication/authorization is not obvious in the current web/API boundary; production should require user identity and workspace-scoped authorization on API mutations.
  • The API process runs HTTP, schedulers, and workers together. This is simple, but heavy sync jobs can compete with request latency unless separated later.
  • /api-proxy forwards many methods to the API. It should be protected by the same auth model as the dashboard.
  • Queue connection settings favor fast failure when Redis is unreachable. That is good for avoiding hangs, but sync availability depends directly on Redis health.
  • Environment variable naming uses both WEB_APP_URL and FRONTEND_URL in Shopify callback code. These should be aligned or deliberately documented.
  • The root vercel.json is empty while apps/web/vercel.json exists. Deployment ownership should be explicit in Vercel project settings.
  • There are two dashboard generations: current Next.js app and earlier Python dashboard. The Python path should be marked prototype, archived, or maintained intentionally.
